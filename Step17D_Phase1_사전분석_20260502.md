## 0. TL;DR

자동 재로그인 시스템은 **이미 부분 구현되어 있음**. Step 17-D는 신규 구축이 아니라 **3가지 보강 작업**:

1. **능동적 만료 예방** (현재는 ensure_logged_in 호출 시점에만 동작 = passive)
2. **자격증명 보관 격상** (현재 settings.json 평문 → keychain/env)
3. **Hubnet 통합 헬스체크** (`/api/health`에 hubnet 세션 상태 누락)

## 4. 세션 만료 감지 메커니즘

### 4-1. KREAM 판매자센터 (passive)
```
ensure_logged_in(page, context)
  → page.goto(/c2c)
  → if "/sign-in" in URL: _try_auto_relogin() 호출
  → if 빈 페이지(body_len < 20): _try_auto_relogin() 호출
```
**한계**: 호출 시점에만 동작. 30분/1시간 단위 사전 점검 없음.

### 4-2. KREAM 일반 사이트
- `kream_collector.py`에서 사용. 만료 시 자동 재로그인 흐름 **없음** → 수집 실패 후 alert만.

### 4-3. 허브넷 (active)
```
ensure_hubnet_logged_in()
  → load_hubnet_session() → _is_session_alive(sess) ping
  → ping 실패 시 hubnet_login() 후 save_hubnet_session()
```
**한계**: API 호출 시에만 ping. 사전 갱신 스케줄러 없음.

### 4-4. /api/health
- `auth_partner` (auth_state.json) 체크: exists + mtime + valid 토큰
- `auth_kream` (auth_state_kream.json) 체크: 동일
- ❌ **`auth_hubnet` 누락** — 헬스체크에 허브넷 세션 상태 없음

## 6. Phase 2 본 작업 설계 제안

### 6.1 자격증명 보관 방식

| 방식 | 장점 | 단점 | 권장 |
|---|---|---|---|
| **macOS Keychain** (`keyring` 모듈) | OS 수준 암호화, 사용자 로그인 시 자동 잠금 해제 | 다른 머신 마이그레이션 시 재설정 필요 | ✅ **1순위** |
| 환경변수 (.env + python-dotenv) | 간단, 12factor 표준 | 평문 (.env), 권한 관리 필요 | 2순위 |
| settings.json 유지 | 현재 흐름 그대로 | 백업 파일에도 평문 노출 위험 | ❌ 비권장 |

→ **Keychain** 권장. settings.json은 "키체인에 없으면 fallback"으로만 사용. 백업 파일에서는 자격증명 키 자동 마스킹.

### 6.2 재로그인 트리거 (3가지 동시 운영)

1. **수동 트리거**: 기존 `/api/session/relogin` 그대로 유지
2. **세션 만료 감지 시 (passive)**: 기존 `ensure_logged_in` → `_try_auto_relogin` 그대로 유지 + 허브넷도 동일 패턴 추가
3. **사전 갱신 스케줄러 (active, NEW)**:
   - 매 6시간마다 `/api/session/status` 체크
   - mtime 18시간 초과 + 토큰 valid 시 사전 재로그인 시도 (만료되기 전)
   - 백업 + 검증 + 덮어쓰기 (성공 시에만)

### 6.3 새 auth_state 저장 안전장치

```
1. 새 로그인 성공
2. context.storage_state() → 메모리 dict
3. 토큰 키 검증 (token_keys 비어있지 않은지)
4. 기존 파일을 .pre_relogin.bak으로 복사 (백업)
5. 임시 파일에 쓰기 (.tmp)
6. 임시 파일을 _is_state_valid()로 재검증
7. os.replace(.tmp → 정식 파일) (atomic)
8. 24h 이상 된 .pre_relogin.bak은 자동 삭제
```

### 6.4 실패 시 알림

기존 `health_alerter.alert()` 사용. 알림 채널 점검:
- 이메일 (SMTP): smtplib 사용. 실패 시 fallback 없음
- 알림 누락 방지: 알림 발송도 try/except로 감싸서 알림 실패 시 stderr 출력

추가: `/api/notifications` 테이블에 `auth_failure` 타입 알림 누적 → 대시보드에서 경고 배너 표시.

## 7. Phase 2 진행 위험 + 회귀 테스트 시나리오

### 7-1. 위험 요소
- **자격증명 마이그레이션 실수**: settings.json → keychain 이전 시 잘못 옮기면 모든 자동 로그인 정지. **반드시 dry-run 먼저**.
- **세션 동시 재로그인**: 사전 갱신 스케줄러와 passive 재로그인이 동시 실행되면 race condition. **threading.Lock 필요**.
- **Gmail IMAP rate limit**: 짧은 시간에 여러 번 OTP 요청 시 차단 가능. **재시도 backoff 필수**.

### 7-2. 회귀 테스트 시나리오 (Phase 2 후 검증)

| # | 시나리오 | 기대 |
|---|---|---|
| RT1 | settings.json에서 자격증명 모두 삭제 → keychain에서만 읽기 | 자동 로그인 성공 |
| RT2 | auth_state.json mtime을 25시간 전으로 위조 → 사전 갱신 스케줄러 발동 | 백업 생성 + 새 세션 저장 + 기존 백업 정리 |
| RT3 | 새 로그인 실패 시뮬레이션 (Gmail IMAP 비밀번호 잘못) | 기존 auth_state.json 유지, 알림 발송 |
| RT4 | 허브넷 세션 24h 경과 + 사전 갱신 | hubnet_login 자동 재호출 + 새 세션 저장 |
| RT5 | passive 재로그인과 active 사전 갱신 동시 트리거 | Lock으로 순차 처리, 중복 호출 0건 |
| RT6 | /api/health 응답에 auth_hubnet 필드 추가 | exists/mtime/valid 모두 표시 |
