## 2. 수정 단계 (진단 결과에 따라)

각 이슈별 수정 방향. 진단 결과에 따라 일부는 "현재 정상 / 수정 불필요"로 판정될 수 있음.

### 2.1 백업 스케줄러 (이슈 1)

수정 시나리오 A: 스케줄러가 정의돼 있는데 등록 안 됨
- BackgroundScheduler에 add_job 누락 → 추가
- 추가 후 schedulers 응답에 `backup: running` 노출 확인

수정 시나리오 B: 스케줄러 자체가 없고 OS cron으로 돌고 있음
- crontab -l 확인
- cron이 멈춘 이유 진단 (권한, 경로, 환경변수)

수정 시나리오 C: 코드에 백업 스케줄러 자체가 없음
- 신규 추가. 매일 1회 (예: 새벽 4시 KST).
- 기존 백업 스크립트가 있다면 그걸 호출하는 식으로

검증:
- 백업 1회 수동 실행 → 성공
- 스케줄러 등록 후 다음 트리거 시각까지 기다리지 말고, 단기 트리거(예: 1분 후)로 시뮬 후 원래 스케줄로 복원
- /api/health의 last_backup_age_hours가 24시간 이내로 떨어지는지 확인

### 2.2 auth_state_kream 갱신 (이슈 2)

진단 결과에 따라:

수정 시나리오 A: auth_partner와 동일한 갱신 로직이 auth_kream에는 누락
- 갱신 함수 호출 추가

- 스케줄러에 추가 (예: 12시간 간격)

수정 시나리오 C: KREAM 일반사이트 접속이 해외에서 차단되는 환경 제약 (인수인계서 v3 §"작업 환경" 참고)

- 갱신 로직 추가 시, 다음 트리거 시각 시뮬

### 2.4 1408 innerHTML 잠복 (이슈 4)

1868 잠복 버그와 같은 패턴이면 같은 방식으로 수정:
```javascript
// AS-IS:
document.getElementById('XXX').innerHTML = ...;

// TO-BE:
const _setHTML = (id, html) => { const el = document.getElementById(id); if (el) el.innerHTML = html; };
_setHTML('XXX', ...);
```

다른 패턴(이벤트 콜백 안에서 해당 탭 미로드 상태로 호출됨 등)이면 그 맥락에 맞게.

검증:
- 새로고침 후 콘솔에 1408 에러 사라짐
- 해당 함수가 호출되는 정상 경로(예: settings 탭 진입)에서는 정상 동작
