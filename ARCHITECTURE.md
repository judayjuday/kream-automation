## 2. 도메인별 상세 구조

### 도메인 A: KREAM 자동화 (✅ 운영 중)

**목적:** 중국 Dewu 소싱 → KREAM 입찰 → 판매 → 정산 자동화

│        ↕                  ↕                  ↕          │

│  │ 자동 재입찰 │    │ 언더컷 방어  │    │ 입찰 정리   │ │

**기술 사양:**
- 출력: 540px(썸네일) / 1080px(상세), 300ppi, Lanczos/Bicubic Sharper
- 로고: 71×12px (주데이 로고)
- 워터마크: 36.56×6.53px, 28% 불투명도, 5단계 배치 로직
- 4코너 알고리즘: 마스크 0겹침 + 최저 엣지밀도 + 최고 대비
- 색상: HSL L>50%면 +3%, 대비 낮으면 #000 폴백
- 파일명: 대표=브랜드명_품번.png / 상세=DET_품번_넘버.png
- R2 URL: https://pub-a6171463d5644d5397d0127a58028498.r2.dev/

**주요 데이터 테이블 (예정):**
- `image_processing_log` (처리 이력)
- `r2_upload_log` (R2 업로드 결과)
- `product_image_mapping` (상품 ↔ 이미지 R2 URL 매핑)

**개발 원칙:**
- 각 프로그램은 **독립 실행 가능하게 모듈화**
- Python + Pillow 기반
- 입력: 폴더 경로 → 자동 처리 → 결과 폴더 출력
- 에러 시 원본 보존, 처리 로그 출력

## 3. 도메인 간 데이터 흐름

### 흐름 1: KREAM 판매 → 통합 대시보드

```
KREAM 판매 발생
   ↓
sales_history 테이블 INSERT (도메인 A)
   ↓

   ↓

   ↓

```

   ↓

   ↓

   ↓
통합 대시보드 주문 탭 갱신 (도메인 D, M4)

### 흐름 3: SSRO CS 답변 자동화

새벽 6시: 상담 수집 스케줄러 (도메인 C, M6)
   ↓

   ↓
답변 초안 자동 생성 (Claude API)
   ↓

   ↓

   ↓
승인 → SSRO에 답변 자동 전송 (Playwright)
```

### 흐름 4: 품절 감지 → 대체 추천 (장기)

```
SSRO 재고 동기화 (도메인 B, 1시간 주기)
   ↓

   ↓

   ↓

   ↓

   ↓
트렌드 점수 계산 + 주데이 검토 큐 등록
   ↓
주데이 승인 (대시보드)
   ↓
[도메인 E] 이미지 편집 자동화 트리거
   ├── Phase 1: 리사이즈/로고/워터마크/GIF
   ├── Phase 2: 파일명 변환 / 폴더 생성 / R2 업로드 / 검증
   └── Phase 3: 사이즈표 / 문구 초안 (Claude API)
   ↓
image_processing_log + product_image_mapping INSERT
   ↓
[도메인 B] SSRO 상품마스터 자동 채움
   ├── 상품명, 가격, 카테고리
   ├── 이미지 URL (R2)
   └── 상세페이지 HTML
   ↓

   ↓
[도메인 D] 통합 대시보드에 신상품 등록 완료 표시
```

## 5. 데이터베이스 설계 원칙

### 단일 DB vs 분리

**결정: 단일 DB (`automation.db`) 유지, 도메인별 테이블 prefix 분리**

이유:
- 크로스 도메인 쿼리 (예: KREAM + SSRO 통합 매출) 자주 필요
- 1인 사업자 규모에서 분리는 오버엔지니어링
- 백업/관리 단순화

### 테이블 명명 규칙

```
도메인 A (KREAM):
  • bid_*, sales_*, my_bids_*, dewu_*
  • 기존 이름 유지 (마이그레이션 비용 큼)

도메인 B (SSRO):
  • ssro_orders, ssro_products, ssro_inventory
  • multi_channel_* (크로스 채널 매핑)

도메인 C (CS):
  • cs_inquiries, cs_drafts, cs_answer_history

도메인 D (대시보드):
  • dashboard_alerts, dashboard_widgets

도메인 E (이미지 편집):
  • image_processing_log, r2_upload_log, product_image_mapping

도메인 F (신상품 크롤링):
  • crawled_products, crawl_sources, trend_scores

공통:
  • settings, notifications, audit_log
```

### 절대 규칙
- **DROP/DELETE 직접 실행 금지** (TEST_ 접두사 데이터 제외)
- **ALTER TABLE 시 NULL 허용** 필수
- **인덱스 명명: `idx_테이블_컬럼` 형식**
- **인덱스 추가 시 IF NOT EXISTS** 필수
- **모든 마이그레이션은 db-migration Skill 참조**

## 7. 주요 안전장치 패턴 (Step 3, 4, 5에서 정립)

### 패턴 1: 다중 안전장치 (N중 안전장치)
```
1. 기본 OFF (사용자 명시적 ON 필수)
2. 토글 시 confirm() 추가 확인
3. 위험 작업은 유예 시간 (Soft Delete)
4. 하루/시간 한도 설정
5. 도메인 격리 (다른 기능 영향 없음)
6. 스테일 체크 (실행 직전 재확인)
```

### 패턴 2: 로그 + 되돌리기
```
모든 자동 실행:
  • 결과를 *_log 테이블에 INSERT
  • action 필드로 결과 분류 (success/skipped_*/failed)
  • snapshot 필드로 원본 데이터 보존 (Undo 참고)
  • 24시간 이상 보관
```

### 패턴 3: API 6 + UI 패널 + 이력 테이블
```
표준 자동화 기능 한 묶음 = 다음 6개 API:
  • GET    /api/<feature>/status      현재 상태/통계
  • POST   /api/<feature>/toggle      ON/OFF 전환
  • GET    /api/<feature>/pending     승인 대기 (있다면)
  • POST   /api/<feature>/cancel      취소 (있다면)
  • POST   /api/<feature>/run-once    수동 1회 실행
  • GET    /api/<feature>/history     이력 조회
```

```

### 격리 원칙
- 각 에이전트는 **자기 담당 영역만 읽고 수정**
- 다른 에이전트 영역 건드리려 하면 **STOP + 보고**
- 에이전트 간 통신은 오케스트레이터 경유
- 컨텍스트 완전 격리 (메인 대화 깨끗 유지)

### 에이전트 정의 형식
```markdown
---
name: <에이전트명>
model: opus | sonnet
tools: [Read, Edit, Grep, Bash, ...]
---

# 담당 영역
- 파일: ...
- 테이블: ...
- API: ...

# 절대 건드리지 말 것
- ...

# 작업 완료 시 필수
- HANDOFF.md 업데이트
- 회귀 테스트 실행
```
